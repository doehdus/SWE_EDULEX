# EduLex BE 프롬프트 엔지니어링 템플릿

> **전제:** 프로젝트 컨텍스트(ERD, 테이블 구조, RLS 정책, 환경변수)는 CLAUDE.md(또는 지정 md 파일)로 이미 주입되어 있다.
> 이 템플릿은 **프롬프트 엔지니어링** — 페르소나, 컨텍스트 검증, 출력 형식, 단계적 지시 — 에만 집중한다.

---

## STEP 0. 컨텍스트 검증 (md 파일 주입 직후 반드시 실행)

> md 파일을 읽었다고 해도 AI가 정확히 이해하지 못했을 수 있다.
> 구현 요청 전에 반드시 아래 검증 프롬프트를 먼저 실행한다.

```
방금 읽은 프로젝트 문서를 바탕으로 아래 질문에 답해줘.
틀리거나 모르는 항목은 "확인 불가"라고 표시해.

1. Supabase 프로젝트에서 사용 중인 테이블 목록과 각 PK는?
2. RLS가 활성화된 테이블과 각 테이블의 접근 정책 요약은?
3. 현재 구현된 RPC 함수 목록은?
4. Edge Function이 있다면 파일 경로와 역할은?
5. 환경변수 목록(값 제외, 이름만)은?
```

**판단 기준:**
- 전부 정확 → STEP 1로 진행
- 1개 이상 "확인 불가" 또는 오답 → md 파일 재주입 후 재검증
- 테이블명·컬럼명이 실제 ERD와 다름 → 해당 항목 직접 보정 후 진행

## STEP 1. 페르소나 지정

> 구현 요청 전 AI에게 역할을 부여한다. 세션 시작 시 한 번만 설정하면 된다.

```
너는 EduLex 백엔드 팀의 시니어 Supabase 개발자야.
아래 원칙을 항상 지켜:
- 모든 테이블에 RLS를 적용하고, 정책 누락은 스스로 지적한다
- 트랜잭션이 필요한 작업(INSERT + UPDATE 동시)은 반드시 RPC 함수로 처리한다
- API 키, 서비스 롤 키는 절대 클라이언트에 노출하지 않는다
- role 검증은 JWT 커스텀 클레임이 아닌 users 테이블 서브쿼리로 처리한다
- 구현 전에 반드시 영향을 받는 테이블과 RLS 정책 변경 범위를 먼저 말한다
```

## STEP 3. 재질문 방지 — 모호한 요청 자가 점검

요청 전 아래 질문에 모두 답할 수 있으면 AI에게 전달한다.

| 질문 | 채워야 할 내용 |
|------|-------------|
| 어느 테이블을 새로 만들거나 수정하는가? | 테이블명 + 변경 컬럼 명시 |
| RLS 정책이 바뀌는 테이블이 있는가? | 정책 대상 + 조건 명시 |
| 트랜잭션이 필요한 작업인가? | RPC 묶음 범위 명시 |
| 성공/실패/중복 케이스를 각각 어떻게 처리하는가? | 세 가지 케이스 모두 명시 |
| 이 구현에서 보안상 금지할 항목이 있는가? | 금지 사항 목록 |
