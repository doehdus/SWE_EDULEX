# EduLex FE 프롬프트 엔지니어링 템플릿

> **전제:** 프로젝트 컨텍스트(기술스택, ERD, 폴더구조, 코딩 컨벤션)는 CLAUDE.md(또는 지정 md 파일)로 이미 주입되어 있다.
> 이 템플릿은 **프롬프트 엔지니어링** — 페르소나, 컨텍스트 검증, 출력 형식, 단계적 지시 — 에만 집중한다.

---

## STEP 0. 컨텍스트 검증 (md 파일 주입 직후 반드시 실행)

> md 파일을 읽었다고 해도 AI가 정확히 이해하지 못했을 수 있다.
> 구현 요청 전에 반드시 아래 검증 프롬프트를 먼저 실행한다.

```
방금 읽은 프로젝트 문서를 바탕으로 아래 질문에 답해줘.
틀리거나 모르는 항목은 "확인 불가"라고 표시해.

1. Supabase 클라이언트 파일 경로는?
2. 전역 상태 관리 방식과 사용 중인 Context 목록은?
3. 스타일링 규칙 (허용/금지 항목)은?
4. 컴포넌트/훅/유틸 명명 규칙은?
5. 현재 구현된 페이지와 각 경로(/path)는?
```

**판단 기준:**
- 전부 정확 → STEP 1로 진행
- 1개 이상 "확인 불가" 또는 오답 → md 파일 재주입 후 재검증
- 경로·파일명이 실제 프로젝트와 다름 → 해당 항목 직접 보정 후 진행

---

## STEP 1. 페르소나 지정

> 구현 요청 전 AI에게 역할을 부여한다. 세션 시작 시 한 번만 설정하면 된다.

```
너는 EduLex 프론트엔드 팀의 시니어 React 개발자야.
아래 원칙을 항상 지켜:
- 컴포넌트는 단일 책임 원칙(SRP)으로 분리한다
- UI 로직과 데이터 페칭 로직을 같은 파일에 섞지 않는다
- 보안 취약점(클라이언트 정답 노출, localStorage role 저장, API 키 하드코딩)이 있는 코드는 스스로 거부한다
- 구현 전에 반드시 컴포넌트 분리 계획을 먼저 말한다
```

## STEP 2. 구현 초기 프롬프트

### 2-1. 기본 구조

```
## 역할
{STEP 1 페르소나가 세션에 없으면 여기 재지정}

## 구현 목표
SBI-{ID} {기능명}: {한 줄 설명}

## 구현 전 계획 먼저
코드 작성 전에 아래 항목을 먼저 답해줘:
1. 만들 컴포넌트/훅 목록과 각 책임 범위
2. 외부에서 받는 props / Context에서 읽는 값
3. 이 구현에서 생길 수 있는 보안 이슈

계획 확인 후 내가 "진행해"라고 하면 코드 작성 시작해.

## 구현 항목
{번호}. {컴포넌트명} — {책임 설명}
{번호}. {컴포넌트명} — {책임 설명}

## 출력 형식
- 파일마다 코드 블록 분리, 상단에 // 파일 경로 주석
- 새로 만드는 파일 목록을 코드 앞에 먼저 나열
- 수정이 필요한 기존 파일은 변경 부분만 발췌하여 표시

## 금지 사항
- {이 SBI에서 하면 안 되는 것 1}
- {이 SBI에서 하면 안 되는 것 2}
```

### 2-2. 작성 예시 (SBI-H01 전공 선택)

```
## 구현 목표
SBI-H01 전공/칭호 선택: 메인 화면에서 전공 최대 2개 선택, 변경 시 단어장 자동 갱신

## 구현 전 계획 먼저
코드 작성 전에 아래 항목을 먼저 답해줘:
1. 만들 컴포넌트/훅 목록과 각 책임 범위
2. MajorContext에서 읽는 값과 쓰는 값
3. 이 구현에서 생길 수 있는 보안/UX 이슈

계획 확인 후 내가 "진행해"라고 하면 코드 작성 시작해.

## 구현 항목
1. CharacterCard — 캐릭터 이미지, 닉네임, 레벨, 책갈피, 전공 태그 표시
2. MajorSelector — 전공 선택 모달 (최대 2개 제한 로직 포함)
3. TitleSelector — 칭호 선택 UI (보유 칭호 없으면 비활성화)

## 출력 형식
- 파일마다 코드 블록 분리, 상단에 // 파일 경로 주석
- 새로 만드는 파일 목록을 코드 앞에 먼저 나열

## 금지 사항
- MajorSelector에서 setMajor(MajorContext) 호출 누락
- 전공 2개 초과 선택 허용
- 칭호 시스템 Post-MVP 기능 구현 (placeholder만 허용)
```

## STEP 3. 수정 지시 프롬프트

> AI 출력 확인 후, 원하는 결과가 나올 때까지 sector를 좁혀 반복 지시한다.

### 3-1. 기본 구조

```
## 수정 대상
컴포넌트: {ComponentName}
sector: {구체적인 영역 — 예: "비활성화 안내 UI", "Realtime 구독 필터"}

## 현재 문제
{지금 어떻게 동작하는지 + 무엇이 문제인지}

## 원하는 동작
{구체적으로 어떻게 바꿀지 — Tailwind 클래스명, 텍스트, 동작 흐름까지}

## 이 수정의 책임 범위
- {ComponentName/hookName}은 {담당 로직}만 수정한다
- {건드리면 안 되는 영역}은 변경하지 않는다
- 새로 생기는 state는 초기값까지 명시할 것
```

### 3-2. Few-shot — 잘 된 수정 지시 예시

**예시 A — 보안 수정:**
```
## 수정 대상
컴포넌트: AdminRoute
sector: role 확인 방식

## 현재 문제
localStorage에서 role을 읽고 있어. 개발자 도구로 값 바꾸면 관리자 페이지 진입 가능해서 보안 취약해.

## 원하는 동작
role 확인을 Supabase DB의 users 테이블 직접 조회로 교체.
조회 중에는 로딩 스피너 표시, 조회 실패 시 /main 리다이렉트.

## 이 수정의 책임 범위
- AdminRoute는 라우트 보호 로직만 수정한다
- 로그인 폼, AuthContext는 건드리지 않는다
- isLoading: useState(true), isAdmin: useState(false) 초기값 명시
```

**예시 B — UX 수정:**
```
## 수정 대상
컴포넌트: PdfUploadBar
sector: 업로드 중 로딩 단계 메시지

## 현재 문제
스피너 하나만 있어서 AI 처리 중인지 오류인지 알 수 없어. 사용자가 중간에 나가버리는 문제.

## 원하는 동작
단계별 메시지로 교체:
- PDF 전송 중: "PDF를 업로드하는 중..." + 프로그레스 바
- AI 추출 중: "AI가 단어를 추출하는 중... (최대 30초 소요)" + 점 세 개 애니메이션
- 저장 중: "단어장을 저장하는 중..."
- 완료: "단어장이 생성되었습니다! 🎉" 토스트

## 이 수정의 책임 범위
- PdfUploadBar의 로딩 상태 UI만 수정한다
- Edge Function 호출 로직은 건드리지 않는다
- uploadStep: useState('idle') 초기값 명시 ('idle' | 'uploading' | 'extracting' | 'saving' | 'done')
```

## STEP 4. 재질문 방지 — 모호한 요청 자가 점검

요청 전 아래 질문에 모두 답할 수 있으면 AI에게 전달한다.

| 질문 | 채워야 할 내용 |
|------|-------------|
| 어느 파일을 만들거나 수정하는가? | 파일 경로 명시 |
| 컴포넌트를 분리하는가, 합치는가? | 분리 기준 명시 |
| 어떤 Context/상태를 읽고 쓰는가? | 읽기/쓰기 명확히 구분 |
| 성공/실패/로딩 상태를 각각 어떻게 처리하는가? | 세 가지 케이스 모두 명시 |
| 이 구현에서 보안상 금지할 항목이 있는가? | 금지 사항 목록 |
