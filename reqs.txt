[REQ-002] Project Vision & Goal
- 대상: 전공 수업/원서를 읽는 대학생, 전공별 핵심 단어 효율 암기 및 실질적 이해 제공
- 게이미피케이션 요소로 지속적 학습 동기 부여하는 웹 환경
- 근거: 게이미피케이션 단어 암기 성공률 = 전통 방식 대비 110%~140%
- 목표: 최대 숙련도 학습 기준으로 140%에 근접한 암기 성공률 달성

[REQ-007] GitHub Kanban 기반 스프린트 운영 방식
- 팀별 item 태깅: Team1/Team2 라벨로 칸반 보드에서 담당 item 분리 관리 (서기가 감독)
- 워크플로우: In Progress(new branch 생성·개발 시작) → Done(F/B 기능 비교 실행, 결합) → Review & Retrospection
- 개발 순서: FE·BE 각자 독립 개발 → 비정기 스크럼(방식·프롬프트 공유) → F+B 결합 → 비정기 스크럼 (서기가 ppt내용준비)
- 브랜치: item 시작 시 new branch 작성, Done 이후 front+back 결합하여 통합 검증
- 브랜치 정책: new branch → develop까지만 merge, main 직접 push 금지

[REQ-008] 산출물 관리 및 협업 도구
- GitHub: 프로젝트 Vision&Goal, 기술스택, 협업규약, 커밋메시지 규칙, 소스코드, DB설계서(ERD), API명세서, WBS
- Notion: 요구사항 정의서, 서비스 기획서, 최종 보고서, 사용자 매뉴얼
- KakaoTalk: 긴급 이슈·알림, 빠른 의사결정(긴급회의)
- Git 규약: 브랜치 전략(new branch → develop, main push 금지), 커밋 메시지 타입 feat/fix/docs/style 준수
- 브랜치 네이밍: 모노레포 기준 파트 명시 prefix 사용 (예: feat/fe-signup, feat/be-db-schema)

[REQ-009] Sprint Flow (7일 주기)
- 1일차 (요구사항 분석·설계): Backlog 선정 및 구체화, API 명세서 픽스, 의존성·우선순위 매핑
- 2~3일차 (병렬 개발): Front/Back 병렬 개발, 비정기 Scrum, 완성 기능 test
- 4일차 (중간 점검): Test 및 Review, Develop 방식·Prompt 공유
- 5~6일차 (통합): Front & Back 결합, UI/UX 조정, 비정기 Scrum
- 7일차 (최종 점검): 최종 Test 및 Review, Sprint 회고, 다음 Sprint 계획 및 역할 순환
